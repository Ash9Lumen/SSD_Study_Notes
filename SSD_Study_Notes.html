Defense in Depth (DiD)

Week 1
What is DiD? A security strategy using multiple, overlapping layers of defense. If one layer fails, others are still in place — prevents a single point of failure. Encompasses physical, technical, and administrative controls (like a castle's layered defenses).
DiD Layers (outer → inner) Policies/Procedures/Awareness → Physical (fences, locks, guards) → Perimeter (firewalls, NAT, DoS prevention) → Internal Network (encryption, VPN) → Host (OS patches, antimalware) → Application (AAA, SSO, coding practices) → Data (DB security, content security, message-level security)
4 Key Principles of DiD Layered Security · Redundancy · No Single Point of Failure · Holistic Protection (people, technology, operations)

SDLC & NIST SSDF Framework

Week 1
SDLC Phases Planning → Engineering → Documentation → Design → Development → Testing → Deployment → Maintenance
NIST SSDF (SP 800-218) Secure Software Development Framework — integrates security into every SDLC stage. Cycle: Plan → Develop → Build → Test → Release → Deploy → Operate. Drawn from 150+ resources including OWASP, SAMM, BSIMM, ASVS.
Best Practices for Secure Implementation Apply 'Secure by Design' early · Follow OWASP guidelines · Regular code reviews & automated scans · Use least privilege + DiD · Secure configs & continuous testing · Train developers in secure coding.
💡 Summary mantra: Secure implementation bridges the gap between secure design and deployment. Ignoring it creates hidden vulnerabilities that expand attack surfaces.

Software Supply Chain & Integrity Attacks

Week 2
What is a Software Supply Chain? All components involved in building software: source code, third-party libraries, package repositories, build tools, and CI/CD pipelines. Each dependency introduces potential risk.
Supply Chain Attack Attackers compromise software before it reaches users by inserting malicious code into trusted components (repositories, updates, or build systems).
Common Attack Techniques Compromised package repositories · Malicious open-source libraries · Dependency Confusion (attacker uploads malicious package with same name — package manager installs attacker version) · Compromised build pipelines
Case Study: SolarWinds (2020) Attackers infiltrated the SolarWinds build system, inserted malicious code into Orion software updates, and thousands of organizations downloaded the compromised update.
Why Hard to Detect? High trust in software updates · Automated dependency installation · Large attack surface · Modifications are subtle and hidden in trusted code.
ControlWhat It Does
Verify package authenticityConfirm packages come from a trusted, unmodified source
Digital signaturesCryptographically prove code hasn't been tampered with
Dependency scanningAutomatically detect known vulnerabilities in libraries
Protect CI/CD pipelinesPrevent injection of malicious code during build process
Least privilege in buildsLimit what the build environment can access/modify

Secure Software Design Principles

Week 2–3
Input Validation & Canonicalization Always Canonicalize first (normalize/decode input into standard form), then Validate. Use allowlists over denylists. Apply contextual checks: length, type, range. Validate after decoding to catch encoded attacks like %2e%2e/ (= ../../).
Canonicalization 3-Step Process 1. Canonicalize — decode/resolve (e.g., %2e%2e → .. via unquote() / os.path.realpath()). 2. Validate — apply allowlist/pattern checks. 3. Process — use the safe, validated data.
Use Well-Tested Libraries Never write custom crypto or parsers. Crypto: use OpenSSL, libsodium, bcrypt — avoid SHA1. Parsing: use yaml.safe_load, JSON schema validation. Auth: use OAuth2/JWT via mature libraries.
Principle of Least Privilege (PoLP) Give every process only the permissions it needs. OS: run as non-root. DB: limited roles (read-only vs write). Cloud: least-privileged IAM roles. Run risky operations in sandboxed environments. Compartmentalize microservices with scoped tokens.
Logging & Alerting Log security events and anomalies. Detect deserialization errors, timing probes, oversized payloads. Protect logs — mask PII/secrets, use centralized logging (ELK, SIEM). Key rule: if it's not logged, it didn't happen; if logged but never reviewed, it's useless.
Secure Defaults — Fail Closed / Deny by Default Systems must start in a secure state. Failures deny access, not allow it. Avoid: open ports, default passwords, 'allow all' rules. Do: deny until explicitly granted, disable debug leaks in production, redirect HTTP→HTTPS, reject invalid input by default.
✅ Design lesson summary: Validate → Restrict → Monitor → Deny. Apply OWASP & SSDF principles throughout.

Multi-Tier Architecture

Lecture 1
Layer vs Tier — KEY DISTINCTION Layer = logical separation of code (same machine). Tier = physical separation across different machines/servers.
Three Common Tiers Presentation Tier – User interface (browser/app). | Application/Logic Tier – Business logic, processes requests. | Data Tier – Database storage and retrieval.
Benefits of Multi-Tier Independent scalability · Enhanced flexibility · Better maintainability · Increased security (data tier isolated from presentation tier).
Data Transfer Protocols Presentation→App: HTTP/HTTPS, RESTful APIs. App→DB: JDBC/ODBC over TCP/IP. Between tiers: CORBA, Java RMI, .NET Remoting, sockets.
Data Transfer Objects (DTOs) & Serialization DTOs carry data between tiers minimizing remote calls. Serialization formats: JSON (lightweight, web APIs) and XML (structured data).
AspectLayersTiers
ConceptLogical separation by functionPhysical separation across machines
LocationSame process/machineSeparate servers/networks
BenefitBetter code organisationIndependent scalability
RelationshipMultiple layers in one tierTier hosts one or more layers

Atomicity & Race Conditions

Lecture 2
Atomicity An operation that is indivisible — either completes entirely or not at all. Classic example: bank transfer (debit AND credit must both happen).
Race Condition Multiple threads access shared data simultaneously; result depends on execution timing → data corruption or security bugs. Example: two threads reading same counter before either updates it.
TOCTOU (Time-of-Check to Time-of-Use) App checks a resource's state, then an attacker changes it before the app uses it. Intersection of race conditions and security. Fix: make Check+Use atomic using locks or transactions.
Mitigations Use mutexes, semaphores, monitors for synchronisation. Use ACID transactions in databases. Use thread-safe data structures. Avoid shared mutable state. Use Static & Dynamic Analysis Tools.
📦

Serialization / Deserialization Vulnerabilities

Lecture 2
What is it? Serialization = converting object → byte stream/JSON/XML (safe). Deserialization = reconstructing object from that data (dangerous if input is untrusted).
Impact Most severe: Remote Code Execution (RCE) — attacker executes commands on server. Also: Denial of Service (DoS), data tampering, privilege escalation.
Root Cause Naively trusting data being deserialized; serialisation libraries invoke object methods during reconstruction which can be exploited.
Mitigations Use allowlists for permitted classes. Prefer JSON/Protobuf with schema validation. Avoid native deserialization of untrusted data.
⚠️ Vulnerable pattern: Object obj = in.readObject(); — deserializing directly from an untrusted socket stream with no validation.
⏱️

Timing Attacks (Side-Channel)

Lecture 2
What is it? Attacker infers sensitive data (passwords, crypto keys) by measuring how long a system takes to respond to different inputs. Different operations take slightly different amounts of time — if that difference is data-dependent, secrets leak.
Side-Channel An unintended physical/logical path leaking info: timing, power consumption, electromagnetic radiation, or even sound.
StrategyHow it helps
Constant-Time AlgorithmsExecution time independent of secret value (e.g., hmac.compare_digest)
Timing JitterAdd small random delay to obscure true timing signal
Response GroupingReport times in buckets (e.g., 0–10ms) to reduce precision
Hardware/OS IsolationMemory isolation, Intel SGX for cloud environments
Standard Library UsageUse validated crypto libs with constant-time implementations built in
⚠️ Vulnerable pattern: if user_input == secret_token: — direct string comparison exits early on mismatch, leaking timing info.
📏

Fixed vs Dynamic Input Length Handling

Lecture 2
The Risk Fixed-length buffers → buffer overflow. Dynamic/unbounded inputs → Denial of Service (DoS). Classic vulnerable code: char buf[256]; gets(buf);
Mitigations Use safe functions: fgets, snprintf. Always validate input length before processing. Enforce size limits on file uploads. Apply fuzz testing and boundary testing.

Developer & Tester Checklists

Lecture 2

👩‍💻 Developer Checklist

  • Use DB transactions and atomic updates
  • Avoid unsafe deserialization of untrusted data
  • Use constant-time operations for comparisons
  • Validate and limit input sizes

🧪 Tester Checklist

  • Concurrency stress tests
  • Fuzz deserialization inputs
  • Boundary and timing tests
  • Static code scan for risky APIs
✅ Key mantra: Always validate input, synchronise shared data, and use secure libraries. Apply OWASP and SSDF best practices in design & implementation.
